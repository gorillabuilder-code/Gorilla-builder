The Auto-Healing Loop

The death of the manual debug cycle. How Gor://a catches, analyzes, and fixes runtime errors autonomously.

The Problem with LLM Code

LLMs are brilliant, but they make mistakes. They miss an import, misspell a variable, or botch a React hook. On standard platforms, a single syntax error results in a white screen of death, forcing you to open the console, copy the error, paste it back to the AI, and beg for a fix.

Gor://a Builder completely automates this workflow with the Auto-Healing Loop.

1. The Interceptor & The Frenzy Vacuum

Inside the WebContainer, Gor://a injects a custom console.error and window.onerror interceptor. If Vite or Node.js throws an exception, it is instantly caught.

However, build tools like Vite often throw dozens of cascading errors at once (e.g., "Failed to resolve import X," followed by 10 component crash logs). To prevent blowing out your token limits, we engineered the Frenzy Vacuum:

  • The orchestrator waits for a 1.5-second window of absolute silence (The Fluid Debounce).
  • If 50 errors fire rapidly, they are batched into a single, deduplicated payload.
  • If the AI is already working on a fix, the buffer overwrites itself, preventing the AI from getting confused by intermediate "hot-reload" errors.

2. The Debugger Agent

Once the error cascade settles, the logs are silently fired to our backend /api/.../log endpoint, awakening the Debugger Agent.

The Debugger is a specialized, low-temperature LLM instructed to do one thing: fix code, no small talk.

  • Target Path Extraction: It uses regex to scan the stack trace and identify the exact file that crashed (e.g., src/components/Header.tsx).
  • Deep Context: It pulls the last 5,000 characters of that specific file, alongside the recent chat history, so it knows what feature it was trying to build when it failed.
  • Surgical Patching: It generates a strict JSON payload with an overwrite_file action, patching the bug without hallucinating new features.

The ALL_CLEAR Signal: Once the fix is applied, the WebContainer monitors the app. If the app runs for 8 consecutive seconds without a single error, the system fires the "ALL_CLEAR" event, dropping the loading overlay and presenting your stable, working application.
