Automated SQL Migrations

The AI acts as your dedicated Database Administrator, executing structural changes via raw SQL.

Why We Avoid ORMs

Many frameworks rely on ORMs (Object-Relational Mappers) like Prisma or Drizzle. While great for human developers, they are notorious "VybeCoding" traps for AI. ORMs require the AI to write a schema file, run terminal commands to generate types, and execute push commands—which frequently fail or get stuck in loop errors inside headless containers.

Gor://a bypasses this fragility. We instruct the Swarm to write Raw SQL.

SQL is deterministic, universally understood by LLMs, and leaves zero room for translation errors.

The Migration Workflow

When you ask the AI to build a feature that requires new database tables (e.g., "Add a table to track customer reviews"), the Supabase Agent Swarm takes over.

  1. Writing the Migration: The Coder agent writes a standard SQL file and saves it to the migrations/ directory in your Virtual File System (e.g., migrations/001_init.sql).
  2. API Execution: The Gor://a backend detects that a .sql file was generated. It immediately pauses the WebContainer boot sequence.
  3. The Remote Query: Using your secure Supabase Management Token, the engine fires the raw SQL directly into the Supabase Postgres instance via their /v1/projects/{ref}/database/query REST API endpoint.
-- Example Migration written and executed by the AI:
CREATE TABLE IF NOT EXISTS public.reviews (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    customer_name TEXT NOT NULL,
    rating INTEGER CHECK (rating >= 1 AND rating <= 5),
    comment TEXT,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

-- Note: The AI will also handle Row Level Security (RLS) 
-- policies automatically if instructed.

Seamless Updates: Once the remote database returns a 200 OK status indicating the tables were created successfully, the Swarm proceeds to write the Node.js Express routes to query those new tables, ensuring perfect synchronization between your DB and your backend.
