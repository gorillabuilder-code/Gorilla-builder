The 402 Error (Out of Credits)

What happens when the tank runs dry, and how Gor://a fails gracefully.

The Death of the Silent Crash

There is nothing worse than an AI generating 90% of a critical file, running out of credits, and silently crashing your entire application. Most platforms leave you with a corrupted codebase and a useless generic error message.

Gor://a Builder intercepts token limits before they destroy your code.

How We Handle the Limit

During the Agent Execution Loop, the Gor://a backend runs a strict enforce_token_limit_or_raise() check before every single file operation. If your balance dips below the required threshold, the engine catches the HTTP 402: Payment Required exception.

  1. The Swarm is Paused: The orchestration engine gracefully halts the Coder agent, preventing it from writing incomplete, corrupted code to the Virtual File System.
  2. The WebContainer Survives: Because the file system wasn't corrupted, your live application in the preview window remains stable and usable.
  3. The UI Alert: Instead of a cryptic console error, the Agent Activity feed injects a beautifully styled warning card: "Usage Limit Reached."

Gateway Propagation: The 402 Error doesn't just protect the IDE; it protects your generated apps. If users of your live SaaS app try to spam the AI chat feature and burn through your Gor://a token limit, the Unified Gateway will return a clean 402 error to your Node.js backend, allowing your app to display a "System Busy" message rather than crashing.

How to Recover

If you hit the 402 Error, you have three options to get the Swarm moving again:

  • Wait for the Reset: Your tokens automatically replenish to 500,000 at the start of your next billing cycle.
  • Gamble for More: Head to the Dashboard and try your luck on the Daily Spin the Wheel.
  • Upgrade to Pro: Unlock 5,000,000 tokens by upgrading to the Premium tier (and don't forget to negotiate with Monke for a discount).
