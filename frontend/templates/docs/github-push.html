Pushing to GitHub

Translating the Virtual File System into a Git Tree and shipping it to your account.

Step 3: The Git Translation Engine

Once your code is optimized for Vercel, it is time to move it from Gor://a's Supabase database into your GitHub account. Because we do not run heavy local Git clients in the browser, we accomplish this via the GitHub REST API.

When you click Create & Push, the Gor://a backend executes the following sequence:

  1. Repository Creation: The engine sanitizes your project name (e.g., "My SaaS App" becomes my-saas-app) and calls the GitHub API to create a brand new, Private repository.
  2. Tree Assembly: The engine fetches all your files from the Virtual File System, strips out any internal Gor://a metadata, and constructs a massive JSON payload representing a "Git Tree".
  3. The Commit: The payload is fired at the GitHub API, creating blobs for every file, assembling the tree, and executing a commit titled "Git Commit via Gor://a Builder."
  4. Branch Updates: The main branch pointer is fast-forwarded to this new commit.

Subsequent Updates: If you go back to the Editor, ask the AI to add a new feature, and return to the Deployment Wizard, the button will now say "Update Repo". Gor://a will skip repository creation and simply push a new commit to your existing main branch, triggering Vercel to rebuild your live site automatically.

What is excluded from the push?

Gor://a explicitly filters out internal metadata folders (like .gorilla/) containing your raw prompt images and Figma JSON ASTs. These are kept private on our servers and are not pushed to your GitHub repository to keep your codebase clean.
