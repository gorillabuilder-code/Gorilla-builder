Autonomous Database Provisioning

Zero-click infrastructure. You ask for a database, the engine builds it while you watch.

The Mid-Chat Provisioning Engine

If you've activated the Supabase toggle or requested a database in your prompt, the Gor://a Builder backend intercepts the request before the AI even starts writing your React code. This is our Mid-Chat Provisioning Engine.

While the UI displays "Provisioning Remote Database...", a massive background operation is occurring in a matter of seconds:

  1. Organization Check: The engine pings the Supabase API to find your default Organization. If you don't have one, it creates one named "Gorilla Apps".
  2. Instance Spin-Up: It generates a secure, randomized 16-character database password and tells Supabase to spin up a new PostgreSQL instance on the Free Tier (usually in us-east-1).
  3. The Status Loop: Creating a cloud database takes time. The engine enters a polling loop, checking the Supabase API every few seconds until the project status returns ACTIVE_HEALTHY.
  4. Key Extraction: Once active, the engine queries the API to extract your anon_key (Publishable Key) and your Project URL.

Environment Variable Injection

The magic happens here: you never have to copy-paste these keys. The orchestration engine takes the anon_key and the database URL and dynamically creates a .env file inside your virtual workspace.

# The engine auto-generates this file in the VFS
VITE_SUPABASE_URL=https://xyz123.supabase.co
VITE_SUPABASE_ANON_KEY=eyJhbGciOiJIUzI1NiIsInR5c...
VITE_GORILLA_AUTH_ID=db-uuid-for-app-auth

Permanent Linkage: The engine also saves the supabase_project_ref directly to your project record in the Gor://a Builder database. This guarantees that if you leave the app and come back tomorrow, the AI will still know exactly which database belongs to your project so it can run future migrations.

Once the database is fully booted and the `.env` file is injected, the engine hands control back to the AI Swarm to begin designing the backend logic.
