Vercel Optimization Phase

The critical step where the AI refactors your Express backend into Vercel Serverless Functions.

The Architecture Conflict

While building your app, Gor://a runs a persistent Node.js Express server on port 3000 inside the WebContainer. However, Vercel does not support persistent, long-running Node servers out of the box. Vercel utilizes Serverless Functions.

If you were to deploy your raw codebase to Vercel, the frontend would work, but every single API call (and your Supabase queries) would instantly return a 404 error.

Step 2: How Gor://a Fixes This Autonomously

When you click Start Optimization in Step 2 of the wizard, a massive, hidden AI workflow is triggered:

  1. Vercel.json Injection: The Gor://a backend instantly generates a vercel.json file and injects it into your project. This file maps all /api/* requests directly to your server.js file, acting as a serverless bridge.
  2. The Refactor Prompt: The backend sends a hidden, high-priority system prompt to the AI Swarm: "We are deploying this full-stack application to Vercel Serverless. Overwrite server.js completely. Do NOT call app.listen(). Vercel requires you to EXPORT the express app instead."
  3. Execution: The Coder agent rewrites the bottom of your server.js file, swapping out the local listening port for an ES module export (export default app;).
// What the AI removes:
// app.listen(3000, () => console.log('Server running on port 3000'));

// What the AI writes for Vercel:
export default app;

Do Not Interrupt: This optimization phase requires the AI to read your server.js and rewrite it. It typically takes about 15 to 30 seconds. Let the loader finish.
